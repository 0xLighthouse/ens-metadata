# Linking a social account to your ENS name

This guide explains how to attach a verifiable social-account proof to an ENS name, what gets stored on-chain, and how anyone can verify it without trusting a central directory. Supported platforms today are X and Telegram; the same scheme extends to any platform with a server-verifiable login by adding a new `<platform>.proof` text record and a validator module on the attester. Platform identifiers use reverse-DNS — `com.x` for X, `org.telegram` for Telegram.

## Mental model

A social proof is a **tagged CBOR envelope** signed by an attester service, written to an ENS text record by the name's owner. Verification is one RPC call plus two local `ecrecover` operations — no IPFS round-trip, no third-party lookup.

There are four pieces:

1. **The attester** — a backend service (Cloudflare Worker at `workers/attester`) that observes both your wallet (via SIWE) and your social account (via OAuth / login widget) in the same session, then signs the envelope with its key.
2. **The envelope** — a tagged CBOR structure with unsigned public metadata (handle, platform, method, timestamp) plus a signed inner payload and signature. Detailed below.
3. **The text record** — `com.x.proof` (or `org.telegram.proof`) on the ENS name, holding the hex-encoded envelope. This is the cheap path everyone reads.
4. **The full proof document** — a richer payload pinned to IPFS, referenced by the `prf` field in the signed payload. Verifiers only fetch this when they want to inspect the upstream evidence.

## Envelope format

The on-chain value is a CBOR-tagged map. The tag enables instant type detection without decoding the full map.

### Detection

```
First byte: 0xDB (CBOR major type 6 / tag, 8-byte additional info)
Tag value:  111525057557094 (0x656e73707266 = "ensprf" as big-endian uint48)
```

Any tool reading the text record can check the first byte to know "this is a tagged CBOR envelope" and dispatch accordingly.

### Structure

```
Tag(111525057557094) {
  v:        3,                          // envelope version
  p:        "com.x",                    // platform (unsigned, for indexing)
  h:        "vitalik",                  // handle (unsigned, for display)
  method:   "privy-linked",             // attestation backend (unsigned)
  issuedAt: 1776275269,                 // creation time, unix seconds (unsigned)
  payload:  <bytes>,                    // canonical dag-cbor of signed fields
  sig:      <bytes 65>,                 // EIP-191 signature over keccak256(payload)
}
```

**Unsigned metadata** (`v`, `p`, `h`, `method`, `issuedAt`) — NOT signed. Exists for indexing and display. Can be read without touching any crypto. The handle can be updated without re-attestation.

**Signed payload** (`payload` bytes, decoded as dag-cbor) — the security-critical fields:

```
{
  v:       3,                            // schema version
  p:       "com.x",                      // platform (signed for replay protection)
  uid:     "0x6fbff7f835de7e...",        // signed uid — see "uid construction" below
  name:    "alice.eth",                  // ENS name (replay protection)
  chainId: 1,                            // EVM chain id (replay protection)
  addr:    "0xd8dA...6045",              // wallet observed via SIWE (staleness check)
  att:     "0x7E5F...9Bdf",              // attester key address (trust check)
  exp:     1798761600,                   // expiry, unix seconds
  prf:     "bafyreigh2akis...",          // reference to full proof document
}
```

`p` appears in both the unsigned metadata AND the signed payload. It's signed for replay protection (prevents replaying a `com.x` proof as `org.telegram`), and duplicated unsigned so indexers can read it without decoding the inner payload.

### Why dag-cbor for the inner payload

The inner payload uses `dag-cbor` (RFC 8949 Core Deterministic Encoding) — map keys sorted by length then bytewise, shortest-form integers, no indefinite-length items. This guarantees `decode(encode(x))` produces byte-identical output across implementations and runs. The signature is over `keccak256(payloadBytes)` — if re-encoding produced different bytes, the hash wouldn't match and verification would fail. JSON has no such guarantee.

The outer envelope uses `cborg` with CBOR tags enabled. The tag prefix is what makes the envelope self-describing.

## uid construction

The `uid` field in the signed payload is NOT the raw platform user id. It's a **signature** over the hash of the platform and uid, produced by the attester:

```
uid = personalSign(keccak256("platform:rawUid"), attesterKey)
```

For example, for Telegram user `12345` on platform `org.telegram`:

```
hash = keccak256(toBytes("org.telegram:12345"))
uid  = attesterKey.signMessage({ message: { raw: hash } })
```

This is a 65-byte EIP-191 signature, stored as a `0x`-prefixed hex string.

### Why a signature instead of a plain hash

A plain `keccak256("platform:uid")` is brute-forceable — Telegram's uid space is ~10 billion sequential integers, enumerable in seconds on a GPU. Signing the hash with the attester key makes it unforgeable: an attacker can't produce a valid signature without the key, regardless of how small the uid space is.

### How consumers verify the uid binding

The consumer (typically an agent) already knows the raw platform uid from chat context. To confirm the on-chain proof binds to that specific user:

```ts
import { keccak256, recoverMessageAddress, toBytes } from 'viem'

// 1. Compute the same hash the attester signed
const hash = keccak256(toBytes(`org.telegram:${knownTgUid}`))

// 2. Recover the signer from the uid signature
const recovered = await recoverMessageAddress({
  message: { raw: hash },
  signature: result.uid as `0x${string}`,
})

// 3. Check it matches the attester address from the signed payload
if (recovered.toLowerCase() === attesterAddress.toLowerCase()) {
  // ✓ The attester endorsed this exact platform:uid pair
}
```

No network call, no attester dependency at verify time. Pure local `ecrecover`.

## Step 1: Create the proof

1. Open the proofs app and connect the wallet that controls the ENS name you want to attest.
2. Sign the SIWE message. This proves to the attester that this session controls this wallet.
3. Link your social account in the same session: X via OAuth or Telegram via the Login Widget.
4. Pick the ENS name and confirm the expiry. The attester signs the envelope and returns the encoded bytes.

Your wallet only signs the SIWE message and the on-chain write transaction. The envelope itself is signed by the attester.

## Step 2: Publish the text record

The encoded envelope gets written to the resolver under `<platform>.proof` as a hex string:

```
alice.eth
  ├── com.x.proof        = "0xdb0000656e73707266..."
  └── org.telegram.proof = "0xdb0000656e73707266..."
```

The `0xdb` prefix is the CBOR tag header — any reader can detect "this is an ensprf envelope" from the first byte. The reverse-DNS prefix groups all records for a given platform under one namespace, and `.proof` marks the record type.

## Step 3: Verify the proof

### Cheap path (SDK)

Read the text record, decode the envelope, verify the attester signature and staleness. No IPFS:

```ts
import { createPublicClient, http } from 'viem'
import { mainnet } from 'viem/chains'
import { addEnsContracts } from '@ensdomains/ensjs'
import { proofVerifier } from '@ensmetadata/sdk'

const client = createPublicClient({
  chain: addEnsContracts(mainnet),
  transport: http(),
}).extend(proofVerifier({
  trustedAttesters: ['0x...'], // the attester key address
}))

const result = await client.verifyProof({
  name: 'alice.eth',
  platform: 'com.x',
})

if (result.valid) {
  console.log('handle:', result.handle)     // from unsigned metadata
  console.log('uid:', result.uid)           // the signed uid (signature)
  console.log('method:', result.method)     // from unsigned metadata
  console.log('expires:', result.expiresAt)
}
```

The SDK's `verifyProof` runs four checks:

| Check | Fails with | Meaning |
| --- | --- | --- |
| Decode the envelope + inner payload | `decode-error` | Bytes are corrupt or not a valid envelope |
| `ecrecover(hash, sig) === payload.att` | `bad-signature` | Envelope was tampered with |
| `payload.att ∈ trustedAttesters` | `untrusted-attester` | Attester is not in your allow-list |
| `payload.exp > now` | `expired` | Claim needs re-issuing |
| `payload.addr === currentEnsOwner` | `wrong-owner` | Name transferred since attestation |

### uid binding check (consumer's responsibility)

The SDK confirms the envelope is structurally valid and signed by a trusted attester. It does NOT check that the uid binds to your specific user — only you know the raw uid. After `verifyProof` returns valid:

```ts
import { keccak256, recoverMessageAddress, toBytes } from 'viem'

const hash = keccak256(toBytes(`org.telegram:${knownTgUid}`))
const recovered = await recoverMessageAddress({
  message: { raw: hash },
  signature: result.uid as `0x${string}`,
})
const isMyUser = recovered.toLowerCase() === attesterAddress.toLowerCase()
```

Without this check, anyone can write a valid `org.telegram.proof` linking some other Telegram account to their ENS name and the SDK will happily say "valid".

### Deep path

Fetch the full proof document from IPFS to inspect the upstream evidence (the OAuth payload, Login Widget data, etc.):

```ts
const full = await client.fetchAndVerifyFullProof(result.cid!, {
  expectedOwner: '0x...',
})
console.log('attestation method:', full.method)
```

## Why this design

- **Tagged envelope, not a flat map.** Unsigned metadata (handle, method, issuedAt) lives outside the signed payload, so it can be updated without re-attestation and read without any crypto. The tag prefix (`0xDB` + "ensprf") makes the format self-describing — consistent with the [Objekt envelope](https://docs.objekt.sh/encryption/envelope) pattern.
- **Signed uid is a signature, not a hash.** Prevents brute-force enumeration of the uid space while remaining locally verifiable via `ecrecover`. The consumer needs only the attester's public key (already in the signed payload as `att`) and the raw uid from their own context.
- **The attester is the trust anchor, not the wallet.** The cheap path learns "a trusted attester independently confirmed both the wallet and the social account" — not just "the wallet asserted it". The wallet's only role is publishing the envelope on-chain.
- **Verifiers configure trust independently.** No global registry. Each consumer ships its own `trustedAttesters` set.
- **`p` is in both layers.** Signed for replay protection (com.x proof can't be replayed as org.telegram), duplicated unsigned for indexing without decoding the inner payload.
- **Canonical dag-cbor for the inner payload.** Deterministic encoding guarantees byte-identical re-hashing across implementations.
- **Layered storage.** The text record carries everything a normal verifier needs. The IPFS document carries the upstream evidence for paranoid verifiers.
