# Linking social accounts to an ENS name

This guide explains how to attach a verifiable social-account proof to an ENS name, what gets stored on-chain, and how anyone can verify it without trusting a central directory. Supported platforms today are X and Telegram; the same scheme extends to any platform with a server-verifiable login. Platform identifiers use reverse-DNS notation (`com.x` for X, `org.telegram` for Telegram) and are stored as parameterized ENS text records under the `social-proofs` key.

## How it works

A social proof is a signed attestation written to an ENS text record by the name's owner. Three pieces are involved:

1. **The attester** — a backend service that verifies that you own your wallet (via SIWE) and that you are able to log into your social media account (via OAuth / login widget), then signs the attestation with its key.
2. **The attestation** — a CBOR-encoded structure containing the platform, handle, a blinded user ID, the ENS name, and a timestamp. The attester signs the inner payload; the name owner publishes the attestation on-chain.
3. **The text record** — a parameterized key on the ENS name (for example `social-proofs[com.x]`), holding the attestation.

Verification requires only the text record and two local `ecrecover` operations.

## Step 1: Create the proof

1. Open the proofs app and connect the wallet that controls the ENS name you want to attest.
2. Sign the SIWE message. This proves to the attester that the current session controls this wallet.
3. Link your social account: X via OAuth, or Telegram via the Login Widget.
4. The attester signs the attestation and returns the encoded bytes.

Your wallet only signs the SIWE message and the on-chain write transaction.

## Step 2: Publish the text record

The attestation gets written to the resolver as a hex-encoded parameterized text record:

```
alice.eth
  ├── social-proofs[com.x]          = "0xda61747374..."
  └── social-proofs[org.telegram]   = "0xda61747374..."
```

Each value starts with `0xDA`, the CBOR tag header, followed by the tag value `1635021684` (`0x61747374`, the ASCII encoding of "atst"). Any reader can check this prefix to identify the value as a social proof attestation.

If you are setting the record manually (rather than through the proofs app), use the "+ Record" button in [ensmetadata.app](https://ensmetadata.app), enter the full parameterized key (for example `social-proofs[com.x]`), and paste the hex-encoded attestation as the value.

## Step 3: Verify the proof

### SDK verification

Read the text record, decode the attestation, and verify the attester signature and ownership using the SDK:

```ts
import { createPublicClient, http } from 'viem'
import { mainnet } from 'viem/chains'
import { addEnsContracts } from '@ensdomains/ensjs'
import { proofVerifier } from '@ensmetadata/sdk'

const client = createPublicClient({
  chain: addEnsContracts(mainnet),
  transport: http(),
}).extend(proofVerifier({
  trustedAttesters: ['0x...'],
  maxAge: 90 * 24 * 60 * 60, // optional: reject proofs older than 90 days
}))

const result = await client.verifyProof({
  name: 'alice.eth',
  platform: 'com.x',
})

if (result.valid) {
  console.log('handle:', result.handle)
  console.log('uid:', result.uid)
  console.log('attester:', result.attester)
  console.log('issued:', new Date(result.issuedAt * 1000).toISOString())
}
```

The SDK's `verifyProof` runs these checks:

| Check | Fails with | Meaning |
| --- | --- | --- |
| Decode the attestation and its payload | `decode-error` | Bytes are corrupt or not a valid attestation |
| Recover the signer from the signature | `bad-signature` | Attestation was tampered with |
| Recovered attester is in the trusted set | `untrusted-attester` | Attester is not in your allow-list |
| Payload address matches the current ENS owner | `wrong-owner` | Name transferred since attestation |
| `now - issuedAt` within `maxAge` (if set) | `stale` | Proof is older than your threshold |

There is no global registry of trusted attesters. Each consumer configures its own `trustedAttesters` set and `maxAge` threshold independently.

### Verifying the uid binding

The SDK confirms the attestation is structurally valid and signed by a trusted attester, but it does not check that the blinded uid binds to a specific platform user. Only the consumer knows the raw platform uid (for example, from a Telegram chat context). After `verifyProof` returns valid, verify the binding locally:

```ts
import { keccak256, recoverMessageAddress, toBytes } from 'viem'

// 1. Compute the same hash the attester signed
const hash = keccak256(toBytes(`org.telegram:${knownTgUid}`))

// 2. Recover the signer from the uid signature
const recovered = await recoverMessageAddress({
  message: { raw: hash },
  signature: result.uid as `0x${string}`,
})

// 3. Check it matches the attester address
if (recovered.toLowerCase() === result.attester.toLowerCase()) {
  // The attester endorsed this exact platform:uid pair
}
```

This check is pure local `ecrecover` with no network call. The uid field and the attestation signature both recover to the same attester address, which is the invariant that ties the platform identity to the on-chain proof.

Without this check, a valid attestation only proves that a user who owned the specified handle at the time also owned access to the wallet. Doing this check confirms the original owner of the handle is still the owner who owns the handle today.

## Attestation format

Each attestation is a CBOR-tagged array identified by tag `1635021684` (`0x61747374`, the ASCII encoding of "atst"). The tag makes the format self-describing: any tool reading the text record can detect the tag and know how to decode the rest.

The attestation contains a version number (`1` for this format), the encoded payload, an attester address, and the attester's EIP-191 signature over the payload hash. The attester address stored in the attestation is an unsigned hint for lookup purposes; the true attester identity is always recovered from the signature.

### Payload fields

The signed payload contains the data that makes up the actual identity claim. Field names are abbreviated to single characters to reduce on-chain storage cost:

| Field | Key | Description |
| --- | --- | --- |
| Platform | `p` | Reverse-DNS platform identifier, e.g. `com.x` or `org.telegram` |
| Handle | `h` | Social handle at the time of attestation. Because this field is signed, a handle change invalidates the proof and requires a new attestation. |
| Blinded uid | `u` | An attester-signed hash of the platform user ID. Consumers who know the raw uid can verify it locally (see "Verifying the uid binding" above). |
| Name | `n` | The ENS name the attestation is bound to, providing replay protection. |
| Issued at | `t` | Unix timestamp in seconds. There is no expiry field; freshness is left to the verifier's `maxAge` policy. |
| Address | `a` | The wallet address observed via SIWE during the attestation session. |

The payload is encoded using dag-cbor (RFC 8949 Core Deterministic Encoding), which guarantees byte-identical output across implementations and runs. This is required because the signature covers the hash of the encoded payload bytes: if re-encoding produced different bytes, verification would fail.
