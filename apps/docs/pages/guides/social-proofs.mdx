# Linking social accounts to an ENS name

This guide explains how to attach a verifiable social-account proof to an ENS name, what gets stored on-chain, and how anyone can verify it without trusting a central directory. Supported platforms today are X and Telegram; the same scheme extends to any platform with a server-verifiable login. Platform identifiers use reverse-DNS notation (`com.x` for X, `org.telegram` for Telegram) and are stored as parameterized ENS text records under the `social-proofs` key.

## How it works

A social proof is a signed attestation written to an ENS text record by the name's owner. Three pieces are involved:

1. **The attester** — a backend service that verifies that you own your wallet (via SIWE) and that you are able to log into your social media account (via OAuth / login widget), then signs the attestation with its key.
2. **The attestation** — a CBOR-encoded structure containing the platform, handle, a blinded user ID, the ENS name, and a timestamp. The attester signs the inner payload; the name owner publishes the attestation on-chain.
3. **The text record** — a parameterized key on the ENS name (for example `social-proofs[com.x]`), holding the attestation.

Verification requires only the text record and two local `ecrecover` operations.

## Step 1: Create the proof

1. Open the proofs app and connect the wallet that controls the ENS name you want to attest.
2. Sign the SIWE message. This proves to the attester that the current session controls this wallet.
3. Link your social account: X via OAuth, or Telegram via the Login Widget.
4. The attester signs the attestation and returns the encoded bytes.

## Step 2: Publish the text record

The user's social media handle is written to their ENS name as a text record. The attestation is written as a separate text record, parameterized by the platform and the attester's address.

```sh
alice.eth
  ├── com.x                                    = "alice"
  ├── attestations[com.x][attester.eth]        = "0xda61747374..."
  ├── org.telegram                             = "alice"
  └── attestations[org.telegram][attester.eth] = "0xda61747374..."
```

## Verifying the proof

To verify a proof that has been published to an ENS name, first collect the following details:

1. The full ENS name (`name.eth`) or subname (`alice.name.eth`) under which the proof was published.
2. The address of the manager of the ENS name (or if the address is a wrapped name, the address of the owner).
3. The ENS name of the attester, and the attestation record itself.
4. Look up which address the attester's ENS name resolves to. This is the signer's address.

## Attestation format

Attestations are encoded using CBOR format, identified by tag `1635021684` (`0x61747374`, the ASCII encoding of "`atst`"), using the following structure:

```sh
Tag(1635021684) [
  2,                                // envelope version
  <integer>,                        // timestamp
  <bytes 65>,                       // signature — EIP-191 over keccak256(payload)
]
```

### Payload

The payload for the signature is encoded using [DAG-CBOR](https://www.npmjs.com/package/@ipld/dag-cbor). The payload can be recreated using the following key names, and the values from the steps listed above:

| Field | Key | Description |
| --- | --- | --- |
| Name | `n` | The ENS name the attestation is bound to. |
| Address | `a` | The wallet address that manages the ENS name. |
| Platform | `p` | Reverse-DNS platform identifier, e.g. `com.x` or `org.telegram` |
| Handle | `h` | The value of the text record matching the platform namespace. (e.g. `alice` for `com.x`) |
| Issued at | `t` | The timestamp from the attestation. |

Generate the DAG-CBOR encoded form of the above fields, and hash it with keccak256 to regenerate the payload that was signed by the attester. You can now verify against the signature included in the attestation.

## Blind UID proofs

A non-public user identifier (like those used internally by social media accounts and exposed during an OAuth flow) can also be attested nd published to the user's ENS name without revealing the private UID. Consumers who already know the user's private UID can verify the proof, but those who do not know it will not be able to reveal it from the attestation.

UID attestations are published to the user's ENS name under the `uid` record, parameterized by the platform and the attester's address.

```sh
alice.eth
  ├── uid[com.x][attester.eth]        = "0xda61747374..."
  └── uid[org.telegram][attester.eth] = "0xda61747374..."
```

These proofs follow the same format as described above, however the payload is slightly different:

| Field | Key | Description |
| --- | --- | --- |
| Name | `n` | The ENS name the attestation is bound to. |
| Address | `a` | The wallet address that manages the ENS name. |
| Platform | `p` | Reverse-DNS platform identifier, e.g. `com.x` or `org.telegram` |
| UID | `u` | The non-public user identifier. (e.g. `12345` for `com.x`) |
| Issued at | `t` | The timestamp from the attestation. |

Using the above details from the user's ENS name, plus the private UID that has been received from an Oauth flow or other method, you can regenerate the signed payload and verify the proof.

## Key rotation

The `attester.eth` segment in every record key is an ENS name, not a raw address. Verifiers resolve it to an address at verify time and check the signature against the current resolution. This gives the attester a simple, on-chain way to rotate signing keys: update the `addr` record on the attester's ENS name, and every signature made by the retired key stops verifying the moment that resolution takes effect. No transaction on the user's ENS name, no revocation list — just an ENS resolver update by the attester operator.
